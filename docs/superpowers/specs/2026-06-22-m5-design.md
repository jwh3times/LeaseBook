## 1. Goal & scope

M5 turns the empty `Reporting` shell into the owner-facing surface of the trust story: a
**statement engine** that renders the complete fiduciary narrative per owner ├Ś property ├Ś period,
**outputs** (print-grade PDF, CSV, and email behind a deferred-send seam), and a **report
catalog** of eight reports with real filters and a live preview.

**In scope (all of M5.1ŌĆō5.3 as one milestone plan):**

- M5.1 statement engine + the structural tie-out + the computed fiduciary panel + consolidated
  multi-property statements + **ADR-016** (the reporting read-layer ADR).
- M5.2 QuestPDF statement template, CSV export, and email delivery **behind a seam** (artifact
  rendered + stored locally; live provider send deferred).
- M5.3 the eight-report catalog with per-report typed filters, builder UI, and live preview.

**Deferred (decided in brainstorm):**

- **Live email send** (the actual SMTP/provider hop) ŌåÆ M8 / when deployed. The delivery record,
  state model, and stored artifact are built and verifiable locally; only the provider send is
  gated. Documented target: **Azure Communication Services Email** (Azure-native fit).
- **Logo upload UI** ŌåÆ M8 polish. The statement PDF white-labels from the PM logo *if present*
  (M2 stored it as a ref) and falls back to PM name/initials; building the file-upload surface is
  not required for the white-labeling to work.

**Out of scope (PRD ┬¦9, rejected on sight):** anything not in the eight-report list; a generic
report-query builder; full GL reporting.

**Boundary rule (ADR-016):** `Modules.Reporting` may cross-schema-read for *non-financial*

assembly half; the no-cross-module-financial-SQL half is the code-review rule ADR-016 records.

## 4. The statement engine (M5.1)

`GetOwnerStatementData(ownerId, propertyId?, period, basis)` returns the `screen-owner.jsx` shape,
computed from the journal over `owner_equity` lines with the standard per-basis filter
(`basis IN (@basis, 'both')`), scoped to the owner (and property when supplied):

| Section | Source |
|---|---|
| Beginning balance | cumulative net owner equity **before** period start |
| Income ŌĆö rent collected | in-period equity credits from `PaymentReceived` / `RentCharged` (per basis) |
| Operating expenses | owner-equity *debits* from `ManagementFeeAssessed` (mgmt fee as **expense to owner**) + `VendorPaid` (maintenance/landscaping/other) ŌĆö *never* `PMIncome` |
| Applied deposits & credits | `DepositApplied` / `PrepaymentApplied` / `CreditIssued` recognized **in the period applied** (the headline fix vs. AppFolio) |
| Disbursement | `OwnerDisbursed` |
| Ending balance | beginning + ╬Ż sections (= owner equity at period end) |

- **`propertyId = null` ŌåÆ consolidated** multi-property statement for the owner.
- Renders in the **org's selected basis** (M2 settings preference), labeled on the header.

### 4.1 Tie-out gate (structural, not cosmetic)

Every in-period `owner_equity` line for the owner (+ property) must land in **exactly one**
section. The engine asserts `ending == GetOwnerLedger balance at period end` ŌåÆ **variance must be
$0.00**. Non-zero variance **blocks issuance** and raises an alert. This makes the PRD "zero manual
adjustments" criterion a code invariant *and* makes any future uncategorized event type fail loudly
the moment it appears on a statement, rather than silently dropping off.

### 4.2 Fiduciary integrity panel = computed assertions

Three checks computed at generation time (never static copy):

1. **PM income excluded Ō£ō** ŌĆö `pm_income` carries no owner dimension, so it is absent from the
   `owner_equity`-scoped query by construction; we additionally assert the result contains no
   `pm_income` line.
2. **Deposits recognized on application Ō£ō** ŌĆö the applied-deposits section reflects in-period
   `DepositApplied` income.
3. **Statement-to-ledger variance $0.00 Ō£ō** ŌĆö the ┬¦4.1 tie-out result.

## 5. Outputs (M5.2)

- **QuestPDF statement template.** Activates the ┬¦1 default ŌĆö register
  `QuestPDF.Settings.License = LicenseType.Community` at startup (free under the $1M-revenue
  threshold; LeaseBook qualifies). Print-grade layout ported from `screen-owner.jsx`: PM-branded
  header, beginning balance, the three sections with subtotals, disbursement line, highlighted
  ending balance, fiduciary panel, the "reconciles to [bank] ŌĆö $0.00 variance" line, **basis
  label**, page numbering. Pixel-reviewed against the prototype.
- **CSV export** of any statement/report via CsvHelper (reuse the `TenantLedgerCsv` pattern).
- **Email behind the seam.** `IStatementDelivery` + a delivery-state model
  (`queued` / `sent` / `failed`) built now. The local impl renders the PDF and stores the
  **immutable artifact** (the exact bytes the owner would receive) via `IArtifactStore` ŌĆö local
  filesystem/Azurite impl now, Azure Blob in the deployed env. Only the live provider send (ACS
  Email) + the `sent` / `failed` transitions are deferred to M8. The artifact + delivery record are
  locally verifiable; only the SMTP hop is gated.

## 6. Report catalog (M5.3)

**Filter model:** per-report *typed* filter sets. Each report is a descriptor declaring its data
source, columns, and accepted filters (period / property / owner / bank / group-by / basis). The
catalog is a registry of descriptors; the builder UI renders only the chips its selected report
declares; the **live preview** re-queries on filter change. No generic query builder.

| # | Report | Data source | New work |
|---|---|---|---|
| 1 | Owner statement | `GetOwnerStatementData` (┬¦4) | **new** (the engine) |
| 2 | All owner ending balances + bank breakdown | `GetOwnerBalances` (exists) | wrap + export |
| 3 | Trust account ledger | bank register read (exists) | wrap + export |
| 4 | Bank reconciliation | M4 immutable snapshots (exists) | render stored snapshot |
| 5 | Security deposit liability | `GetDepositRegister` (exists) | wrap + export |
| 6 | Rent roll | Directory (units/tenants/rent/status) | **new** Directory read |
| 7 | Delinquency (balance aging) | tenant-receivable aging | **new** Accounting read |
| 8 | Management fee income (PM-facing) | `pm_income` lines | **new** Accounting read, explicitly PM-isolated |

Beyond the statement engine: ~3 new reads (rent roll, delinquency aging, mgmt-fee income) +
wrapping 4 existing reads + the builder UI + shared PDF/CSV export plumbing. **Cash/accrual
toggle** on applicable reports; selected basis always rendered on the output. Builder UI ported
from `screen-reports.jsx`: catalog list with category tabs + favorites, builder panel with filter
chips, live preview table, PDF/CSV buttons.

## 7. Testing & the integration gate

**Golden statement fixture (computed-from-seed).** Author the golden owner-statement test against a
real demo owner ├Ś property ├Ś month, asserting every section to the cent. (The specific demo owner
with a clean single-month scenario is chosen and the figures locked during plan-writing ŌĆö see open
item O1.) Replaces trusting the static `PF.statement` mock.

**New invariants (extend the M1.4 harness ŌĆö accounting-adjacent ŌåÆ suite is mandatory):**

- **PM-income exclusion at the statement surface** ŌĆö no `pm_income` line reachable by
  `GetOwnerStatementData`; by construction + a query-result assertion over generated data
  (M1.4 invariant #3, re-pinned to the statement query).
- **Tie-out is property-based** ŌĆö over the existing CsCheck random-event harness, every generated
  statement ties out to its owner-ledger balance ($0.00) and no equity line is uncategorized.
- **Consolidated == ╬Ż per-property** statements for an owner.
- **Report-layer isolation** ŌĆö the mgmt-fee-income report shows `pm_income`; no owner-facing report
  does.

**Output tests:** PDF renders and contains the basis label + key totals (content assertion via
QuestPDF's document model, not just "didn't throw"); CSV round-trips figures; delivery creates a
record + stores the artifact to the local store with state `queued`.

**e2e (Playwright):** run an owner statement ŌåÆ tie-out passes ŌåÆ export PDF/CSV; open the catalog,
change a filter, preview updates.

**┬¦D-style integration gate (same shape as M4's), plus process notes:**

- **M5 adds migrations** (delivery records, artifact metadata, any saved report-filter presets) ŌĆö so
  unlike M3 it is **not** schema-stable. Every new table is org-scoped ŌåÆ must go through the RLS
  helper and pass the schema-guard test. The gate verifies migrate-from-blank.
- **`npm run format:check` joins the per-WP gate** (next to `dotnet format --verify-no-changes`) ŌĆö
  the m4_retro #6 process fix; prettier-dirty web files passed every local M4 gate and only failed
  CI.
- **Audit on money-touching paths** ŌĆö statement issuance + delivery emit audit events (DoD).
- **Gate tail:** seed├Ś2 idempotent ┬Ę `check-invariants` exit 0 ┬Ę full `dotnet test` green ┬Ę web
  lint/typecheck/test/build ┬Ę `schema.d.ts` no-drift ┬Ę docker full-stack serves `/reports` + a
  statement PDF ┬Ę `dotnet format` clean ┬Ę **ADR-016 recorded** ┬Ę `docs/accounting.md` gains a
  statement/reporting section ┬Ę `private/TODO.md` ┬¦M5 checked.
