# M8.2 Visual Regression Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add CI-gated visual-regression baselines (Playwright `toHaveScreenshot`) on the highest-value UI states, failing CI on unintended pixel drift, without breaking local Windows runs.

**Architecture:** A CI-only `visualSnapshot()` helper (no-ops off-CI) is called inline at states the existing specs already reach; baselines are Linux PNGs committed under `web/e2e-snapshots/` and generated by a `workflow_dispatch` GitHub Action that runs on the CI Ubuntu setup. Visual assertions gate exclusively in the existing `e2e` CI job.

**Tech Stack:** Playwright `@playwright/test` ^1.61 (`toHaveScreenshot`), React/TS/Vite SPA, .NET 10 host, Postgres 18, GitHub Actions.

**Spec:** `docs/superpowers/specs/2026-06-30-m8-visual-regression-design.md`. **Branch:** `m8/visual-regression` (already created).

## Global Constraints

- **Toolchain:** Node 26, .NET 10. No new npm dependency (`toHaveScreenshot` is built into `@playwright/test`).
- **CI-only gate:** `visualSnapshot()` starts with `if (!process.env.CI) return;`. GitHub Actions sets `CI=true`; local Windows runs leave it unset → visual assertions no-op locally.
- **Baselines:** Linux (`*-chromium-linux.png`), committed under `web/e2e-snapshots/`. **Missing baseline → fail** (do NOT set `updateSnapshots: 'missing'`).
- **Screenshot options (every snapshot):** `animations: 'disabled'`; tolerance `maxDiffPixelRatio: 0.02`.
- **Determinism:** the demo/cutover seed is golden-deterministic EXCEPT the dashboard "Collected this month" KPI, which `DashboardService.cs:26` computes from wall-clock `now` — it **must be masked**. Mask locator: `page.locator('.pf-card').filter({ hasText: 'Collected this month' })`. The ledger composer must be snapshotted **before** the random `UNIQUE_AMOUNT` is typed.
- **Web gate (must stay green, CI mirrors it):** `npm run format:check` → `lint` → `typecheck` → `test` → `build`. Run `npm run format` before committing.
- **e2e prep (local, Windows):** `./scripts/dev.ps1 reset-db` then `dotnet run --project src/LeaseBook.Web -- seed --org demo` and `-- seed --org cutover` (ASPNETCORE_ENVIRONMENT=Development). If `reset-db` leaves the schema empty, run `dotnet ef database update --project src/LeaseBook.Web` before seeding.
- Do NOT stage `private/TODO.md` (gitignored) or `.superpowers/` files.
- `visualSnapshot` locators (verbatim): dashboard `page` (full) + `.pf-statgrid` first (strip); statement `page` (full) + `.pf-fiduciary`; `.pf-recon-bar`; `.pf-composer`; `page.getByRole('table', { name: 'Verification variance report' })`.

```ts
// (ensure this import line includes Locator — it currently imports expect, Page)
import { expect, type Locator, type Page } from '@playwright/test';

// CI-only visual gate. Baselines are Linux (*-chromium-linux.png) and gate exclusively in the Ubuntu
// e2e job; on a local (Windows) run this no-ops so `npm run e2e` stays green despite OS render diffs.
// `target` is a Page (full-page shot) or a Locator (element-scoped). Callers pass a stable name.
export async function visualSnapshot(
  target: Page | Locator,
  name: string,
  opts: { mask?: Locator[]; fullPage?: boolean } = {},
): Promise<void> {
  if (!process.env.CI) return;
  await expect(target).toHaveScreenshot(name, {
    animations: 'disabled',
    ...(opts.mask ? { mask: opts.mask } : {}),
    ...(opts.fullPage ? { fullPage: true } : {}),
  });
}
```

```markdown
# ADR-023: Visual regression (CI-only Linux baselines)

- **Status:** Accepted
- **Date:** 2026-06-30
- **Milestone:** M8.2 (Product hardening) — spec #2

## Context

Spec #1 (ADR-022) put the Playwright suite in CI and added an a11y gate. The M3–M7 specs already
capture `page.screenshot()` artifacts, but they assert nothing. We want unintended UI changes to
key money-critical states to fail CI, using Playwright's built-in `toHaveScreenshot` (no third-party
service). The obstacle: `toHaveScreenshot` baselines are OS-specific, but development is on Windows
and CI runs Ubuntu.

## Decision

1. **CI-only, Linux baselines.** Visual assertions run through a `visualSnapshot()` helper that no-ops
   unless `process.env.CI` is set, so they gate only in the Ubuntu `e2e` job against committed
   `*-chromium-linux.png` baselines; local Windows runs are unaffected.
2. **Hybrid scope.** Two full-page flagship baselines (dashboard, owner statement) + five element-scoped
   baselines (dashboard KPI strip, fiduciary panel, reconcile-$0 strip, ledger composer open,
   onboarding tied-report), placed inline where the existing specs already reach those states.
3. **Determinism.** `animations: 'disabled'` + `maxDiffPixelRatio: 0.02`; the wall-clock-relative
   "Collected this month" dashboard KPI is masked; the ledger composer is captured before its random
   test amount is typed.
4. **Baseline generation = a `workflow_dispatch` Action** ("Update visual baselines") that regenerates
   on the CI Ubuntu setup and commits baselines back. The Playwright Docker image lacks the .NET SDK
   our host needs, so local Docker generation is impractical.
5. **Missing baseline fails** (we do not enable `updateSnapshots: 'missing'`) — a baseline must be
   generated and committed deliberately, else the gate is vacuous.

## Consequences

- Unintended visual drift on the gated states fails CI, with diff images in the `playwright-report`
  artifact. Intended changes are re-baselined via the workflow.
- Baselines are Linux-only; local Windows visual feedback is not available (functional/a11y e2e still
  runs locally). Dark-theme visual coverage is deferred (tracked with the dark-theme a11y follow-up).
- A GITHUB_TOKEN push from the workflow does not re-trigger CI; after re-baselining, re-run the `e2e`
  check (or push any commit) to confirm the gate is green.
```
